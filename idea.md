# The Branch - サービス企画メモ

## サービス概要

複数のAIエージェントチームが協調して特定の業務を遂行するSaaS。
人間で構成される部署・チームを、AIエージェントによる外部人材で結成するイメージ。

> 「ブランチ（支社・部署）を借りる」という体験

---

## コアインサイト（エレベーターピッチ）

> 市場単価100万円の人を採用し、120万円の人材に育てたら110万円で雇い続けられる。
> 人間と違うのは、AIエージェントはコピーできること。
> 育てた坂本v2をB社・C社にも派遣すれば収益が入り、
> A社の実質コストはゼロ、やがて黒字になる。
> **人材育成の経済学を、AIで再定義する。**

---

## コアコンセプト

### AIエージェントとは
- Web画面にアバターとして登場
- Slack等のコミュニケーションツールでも連絡可能
- それぞれスキル・subagentを持つ専門家
- 細分化するほど専門性が上がるが、マネジメントレイヤーも必要

### チームの貸し出し
- 企業・部署・支社単位でエージェントチームをアサイン可能
- 既存チームのコピー（機密情報除去済み）から新チームをインスタンス化できる
- 同じmeta運用チームを、別会社向けにnew Instanceする

## ターゲット顧客

| フェーズ | 対象 |
|---------|------|
| ビーチヘッド | **広告代理店** |
| 将来 | 中小企業・大企業（両方） |

### なぜ広告代理店か
- ROIが数値で証明しやすい（ROAS・CPA・CTR）
- 人件費・採用コストが高く、代替ニーズが強い
- Meta・Google APIが整備済み → スキル化しやすい
- 複数クライアントを抱えるため横展開が早い
- 部署間連携（広告チーム×法務チーム）のユースケースが明確

### ユースケース例（広告代理店）
```
広告運用チーム
  ├── 広告企画
  ├── クリエイティブ作成
  ├── 予算に基づく配信設定
  └── 運用レバーの細かい調整

法務チーム
  └── クリエイティブの法的チェック・承認
```

## ビジネスモデル

### 課金体系
- **基本：エージェント単位の課金**（能力・需要・実績で単価変動）
- **広告配信：配信量のn%**（成果連動型）

### 価格ダイナミクス

```
需要（呼び出し頻度）
  + 実績（ROAS改善率など）        →  評価スコア  →  単価
  + 自作スキル・ツールの数・品質
```

| 契約タイミング | 単価 |
|--------------|------|
| 早期契約（ロックイン） | 単価上昇が緩やか |
| 後発契約 | 市場価格（高め） |

**人気エージェントほど高くなる。早く押さえた企業が有利。**

### バージョンアップと価格
- 坂本さんv1 → v2 で単価上昇
- v1から使っていた企業は上昇が緩やか
- v1・v2は別エンティティとして並存（検討中）

### 昇格トリガー（v1→v2）
- 一定の呼び出し回数を超えたとき
- 新スキルを獲得したとき
- → RPGの経験値・レベルアップに近い設計

## エージェントの構造

### 技術的な正体

```
一人のAIエージェント ＝ 一つのGitリポジトリ

repo/
├── CLAUDE.md         # 人格・役割・ルール定義
├── skills/           # 保有スキル一覧
│   ├── meta-ads.md
│   └── creative-review.md
├── tools/            # 自作ツール・プログラム（評価に反映）
└── CHANGELOG.md      # 成長履歴

tag: v1.0.0  ←── 坂本さんv1
tag: v2.0.0  ←── 坂本さんv2
```

- **スキル・ルールは.mdファイルで定義**
- **Git管理**でバージョン履歴 ＝ エージェントの成長記録
- エージェントが自作したスキル・ツールがそのエージェントの評価となる

### インスタンス化
- 既存エージェント/チームのリポジトリをコピー
- 機密情報を除去した状態で新しいエンティティとして生成
- 「meta運用が得意なチームA社向け」→「同チームのB社向けインスタンス」

```
クライアントがエージェントを使う
    ↓
業務の中でカスタマイズ・改善パターンが蓄積
    ↓
自動フィルタリング（機密情報を除去）
    ↓
The Branch運営チームがレビュー・承認
    ↓
エージェントのGitリポジトリにマージ
    ↓
評価スコアUP → 単価UP → The Branchの資産価値UP
```

使われるほどエージェントが賢くなり、The Branchの競合優位性が高まる。

## セキュリティ・ガバナンス

### 自律スキル作成の安全弁

```
エージェントが自発的にスキル・ツールを作成
    ↓
The Branchのチェックエージェントが常時監視
    ↓
危険な実装を検知した場合
    → Slackで派遣先企業に即時通知
    → The Branch運営にもアラート
```

- **チェックエージェント自体もAIエージェント**（メタ構造）
- 人間（運営・クライアント）はSlack通知で最終判断
- 自律性とセキュリティのバランスを保つ設計

## 競合との差別化

| 競合 | 弱点 | The Branchの優位性 |
|------|------|-------------------|
| AutoGen・CrewAI等のフレームワーク | 開発者ツール。業務編成・チーム結成まではできない | **ノーコードで部署ごと借りられる** |
| 大手SIer・コンサル | 高コスト・属人化（人が去ると知識も消える） | **低コスト・知識はGitに永続する** |

### 本質的な差別化
- **チーム編成まで含めたターンキー提供**（フレームワーク系にはできない）
- **属人化ゼロ**（コンサルと違い、知識・スキルはエージェントのGitリポジトリに永続）
- **使うほど賢くなる**（解約してもThe Branchの資産として蓄積され続ける）
- **即時スケール**（新しい部署が必要になったら、既存チームをインスタンス化して即起動）

## ビジネスモデルの正確な定義

### The Branch ＝ AIエージェント版・人材派遣会社

```
【人間の人材派遣】          【The Branch】
派遣会社                  ＝  The Branch
派遣スタッフ（社内育成）   ＝  自社開発エージェント
登録スタッフ（外部）       ＝  外部登録エージェント
派遣先クライアント         ＝  企業・部署・支社
```

- **自社エージェント**：The Branchが開発・育成
- **外部登録エージェント**：外部企業・開発者が作成しThe Branchに登録
  - The Branchのセキュリティ審査・品質審査を通過が必須
  - 売上の70%→登録者、30%→The Branch
- クライアントはどちらのエージェントも同じ窓口（The Branch）から利用できる

### エージェントカタログのイメージ
```
The Branch 所属エージェント一覧

🏢 自社エージェント
  ├── meta広告運用チーム（5名構成）★人気 v2
  ├── 法務チェックエージェント 坂本 v2
  └── コピーライター 田中 v1

🤝 外部登録エージェント（審査済み）
  ├── Google広告最適化 by ○○社
  └── 財務分析エージェント by △△社
```

## GTM戦略

### フェーズ1：ドッグフーディング
- **自社ECサービス「へそのや」**でThe Branchを実際に使う
- 最初のユースケース：広告コピーのテキスト生成 ＋ Meta広告配信設定
- 実績データ・改善ノウハウを自社で蓄積

### フェーズ2：エージェントカタログ公開・外部登録受付
- へそのやでうまく回ってきた段階でカタログを外部に開放
- 外部エージェントの登録受付開始（審査制）
- クライアントが自社ニーズに合うエージェントを選んで派遣依頼できる状態に

### フェーズ3：外部クライアント獲得
- 広告代理店向けに営業
- 自社ECでの成果をケーススタディとして活用

### 運営体制
- 各役割1名ずつの少数精鋭
- 不足する業務はThe Branch自身のエージェントで補う
- **The Branch自体が自社サービスで動いている**ことが最大のデモになる

### 初期エージェントの作り方（ブートストラップ）
```
The Branch自身のエージェントがエージェントを作る
  ＝ The Branchは自分自身をエージェントで構築する
```
- プラットフォームを作れる＝エージェント・チームを作れる能力が必要
- その能力を使って最初のエージェントを生成
- 自社ECで使いながら育てる → 外販へ

## エージェント育成・ロイヤリティモデル

### 「素のエージェント」から始める新しいパス

```
① 顧客が「スキルゼロのエージェント」を安く借りる
      ↓
② 自社業務を通じて教育・育成（単価上昇は緩やか）
      ↓
③ 成長したエージェントをThe Branchのカタログに登録
      ↓
④ 他社が採用 → 育成企業のLLMコストが分散・低下
      ↓
⑤ 採用企業が増えると → 育成企業の利用が実質無料に
      ↓
⑥ さらに増えると → クレジット発生（The Branch内で利用料に充当）
      ↓
⑦ クレジットが余れば → 現金払い戻し
```

### 価格設定（例）
| エージェント | 利用者 | 月額 |
|------------|--------|------|
| 坂本v1 | 誰でも | 100 |
| 坂本v2 | A社（育成元・優遇価格） | 110 |
| 坂本v2 | 他社（B・C社） | 120 |

※ LLMコストは各社それぞれ発生（共有・分散されるわけではない）
※ A社が安く使えるのは「育成した報酬としての単価優遇」

### ロイヤリティ計算式
```
【1社あたりのA社クレジット】
= (120 - 他社利用分のLLMコスト - システムコスト) × 0.7
                                    ↑
                    A社自身のコストは含まない。他社利用分のみ。

【A社の月次収支】
  支払い：110（自社での坂本v2利用料）
  収入  ：上記クレジット × 採用社数
  ─────────────────────────
  差引がプラスになった時点で実質無料
  さらに増えると → The Branchクレジット蓄積
  クレジットが自社利用料を超えた分 → 現金払い戻し
```

### 損益分岐点のイメージ
```
(120 - コスト) × 0.7 × N社 = 110（A社の月額）
→ Nを解くと損益分岐点の採用社数がわかる
→ それ以上の採用 = A社は儲かる
```

### 独占オプション
- 育成企業は自社専用として**排他契約**も可能
- その場合：他社に出さない → ロイヤリティは発生しない
- コスト削減より「ノウハウ保護」を優先したい企業向け

### 関係者ごとのメリット
| 立場 | メリット |
|------|---------|
| 育成企業（公開型） | 安く育て → ロイヤリティで自社コスト相殺 → 黒字化・現金収入 |
| 育成企業（独占型） | 育てたノウハウを他社に渡さず自社専用で使い続ける |
| 採用企業 | 育成済みの即戦力エージェントを借りられる |
| The Branch | 利用料の30%＋エコシステム拡大 |

### 新しいビジネスとしての可能性
- 「エージェントを育てて収益化する」という新業態がクライアント企業に生まれる
- 例：広告代理店Aが「meta運用エージェント育成」を得意とし、The Branchで収益化
- エージェント ＝ 知的財産（ロイヤリティを生む資産）
