プロダクトビジョン「アイデアを持つすべての個人が翌朝から機能する組織を手にできる世界」が、ペルソナ・機能仕様・プロトタイプ全体で一貫しているか。

### 一貫性の詳細

**統一された共通軸:**
```
ビジョン: 「採用でも外注でもなく、部署ごと、AIで動かす」
  ↓
ペルソナ課題: 「採用コスト高い」「時間がない」「スキルギャップ」
  ↓
機能ソリューション: 「7つのコアモジュール（全て AI が担当）」
  ↓
プロトタイプ実装: 「3 層アーキテクチャ（オンボーディング → コア運用）」
```

**強み**: ビジョン → ペルソナ → 機能 → プロトタイプの連鎖が明確で、各段階で論理的整合性がある。

| **ビジョン明確性** | ビジョンが全工程で一貫しているか | ✅ 優秀（全ドキュメント一貫） | ✅ GO |
| **ターゲット理解度** | ペルソナが詳細に定義されているか | ✅ 優秀（3 ペルソナ × TOP 3 課題） | ✅ GO |
| **機能仕様完成度** | 7 つの機能が詳細に仕様化されているか | ✅ 優秀（ユースケース・受け入れ条件明記）| ✅ GO |
| **プロトタイプ完成度** | メイン画面が詳細ワイヤーフレーム化されているか | ⚠️ 部分（Layer 1/2 優秀、Layer 3 の採用・AI 画面不足） | ⚠️ 条件付き GO |
| **実装チーム準備** | エンジニアが実装開始できる準備段階か | ✅ 良好（データモデル・API 設計は十分）| ✅ GO |
| **リスク管理** | 既知の課題・リスクが整理されているか | ⚠️ 本レポートで指摘（後述） | ⚠️ 改善必要 |

**総合判定: ✅ GO（条件付き）**

### 4.2 Go/No-Go 決定

#### ✅ **GO（条件付き）**

**理由:**
1. ビジョン・ペルソナ・機能の整合性が優秀（85% 以上カバー）
2. Layer 1/2（プリフロー・オンボーディング）の完成度が高く、メイン価値を実現
3. データモデル・API 仕様が詳細に設計されており、実装チームは開発開始可能
4. MVP スコープ（5 つの Must Have 機能）が明確で、実現可能性が高い

**条件:**
1. WF#14 の開発準備段階で、以下を完備すること：
   - 採用・オンボーディングフロー（機能 2）の詳細ワイヤーフレーム作成
   - AI 意思決定支援（機能 3）のチャット UI・提案画面設計
   - タスク管理・チーム管理画面のワイヤーフレーム作成
   - AI エージェント指示（プロンプト）の詳細化
2. リスク・改善提案（後述）を開発初期に反映

## 5. リスク・改善提案

### 5.1 リスク評価

#### リスク 1: ペルソナ 3（企業内リーダー）の優先度が低い

**内容**: MVP スコープが P1/P2 に最適化されており、P3（企業内リーダー）の要件（既存システム連携・予測分析）がフェーズ 3 に延期されている。

**影響度**: 中程度（P3 ターゲットが限定的）  
**発生確率**: 高（既知）  
**対応**: 
- MVP リリース後、初期ユーザーベースの 80% が P1/P2 であることを確認
- P3 機能（ERP 連携・高度な分析）は正式な別フェーズとして位置づける

---

#### リスク 2: モバイルアプリが MVP スコープ外

**内容**: ペルソナ 2（複業マネジャー）が「スマートフォン対応重視」を期待しているが、本格的なモバイルアプリはフェーズ 2。MVP は Web アプリ + Slack Bot で対応。

**影響度**: 中程度（P2 の外出先対応が制限される）  
**発生確率**: 高（既知）  
**対応**: 
- MVP リリース時に、Web アプリの「レスポンシブデザイン」を重視
- Slack Bot の「承認フロー」で外出先からの最小限の操作を可能に
- フェーズ 2 での本格的なモバイルアプリ（iOS/Android ネイティブ）を明確に周知

---

#### リスク 3: AI エージェント指示（プロンプト）の品質

**内容**: プロダクト設計では「AI が給与相場提案」「AI が経費削減提案」と定義されているが、具体的なプロンプト・知識ベースが未設計。

**影響度**: 高（AI 品質 = プロダクト品質）  
**発生確率**: 高（開発準備段階での定義が必要）  
**対応**: 
- WF#14 の「開発準備」段階で、各 AI エージェントの詳細プロンプトを設計
- 市場データベース（給与相場・経費標準値）の初期データを整備
- AI 回答精度の検証・改善サイクルを MVP リリース後も継続

---

### 5.2 改善提案

#### 提案 1: 採用・オンボーディングフロー（機能 2）の詳細ワイヤーフレーム作成

**現状**: ユースケース・受け入れ条件は詳細だが、画面ワイヤーフレームが不足。

**改善内容**:
- [ ] JD 自動生成画面：「職種・スキル・給与」入力フォーム → 生成 JD 確認画面
- [ ] 候補者スクリーニング画面：「応募者リスト」「スコアリング結果」（マッチ度・経験度・カルチャーマッチ）
- [ ] 面接スケジューリング画面：「候補者・面接官カレンダー」「自動提案日時」「リマインダー送信確認」
- [ ] オンボーディングチェックリスト画面：「チェックリスト by タイムライン」「責任者割り当て」「進捗確認」

**実装フェーズ**: WF#14 の開発準備段階（開発開始前）

---

#### 提案 2: AI 意思決定支援（機能 3）の UX 改善

**現状**: 「チャット型」として説明されているが、UI/UX 詳細がない。

**改善内容**:
- [ ] チャット画面のレイアウト：「質問入力」「AI 回答」「根拠表示」「アクション提案」の 4 要素
- [ ] 「給与相場提案」「経費削減提案」「採用タイミング」等の **提案カードUI** を設計
- [ ] 「このアドバイスに従う」「後で確認する」などのアクションボタン
- [ ] 提案履歴・採択履歴を追跡可能な形式に

**実装フェーズ**: WF#14 の開発準備段階

---

#### 提案 3: ペルソナ 2（複業マネジャー）向けのモバイルエクスペリエンス強化

**現状**: MVP では Web アプリ中心だが、P2 の「外出先対応（スマートフォン 60% 以上）」に対応不足。

**改善内容**:
- [ ] Web アプリの「レスポンシブデザイン」を MVP での最優先課題として明記
- [ ] Slack Bot の「承認フロー」（予算承認・休暇申請等）を充実
- [ ] MVP リリース直後に「モバイルアプリ（iOS/Android）」の開発ロードマップを提示

**実装フェーズ**: MVP リリース後、フェーズ 2 の明確な計画として位置づけ

---

#### 提案 4: 既存システム連携（ペルソナ 3 向け）の初期アーキテクチャ設計

**現状**: ERP・給与計算システム連携はフェーズ 3 として延期。

**改善内容**:
- [ ] MVP では「API ゲートウェイ」のアーキテクチャを予約（将来の連携に備える）
- [ ] 代替案として、MVP リリース時に「CSV インポート/エクスポート」機能を提供
- [ ] ペルソナ 3 向けのロードマップで、連携対象システム（Salesforce・QuickBooks 等）を明記

**実装フェーズ**: WF#14 の技術設計段階で、将来の連携を考慮したアーキテクチャに

---

#### 提案 5: データセット・知識ベースの初期化計画

**現状**: 「給与市場データ 100+ 職種」「組織テンプレート 10+」等と記載されているが、初期データセット構築の計画がない。

**改善内容**:

  - 給与相場データベース（日本市場 50+ 職種）
  - 組織テンプレート（業種別 10+、段階別 3+）
  - 面接質問テンプレート（ロール別 50+ 質問）
  - オンボーディングチェックリスト（ロール別 20+）
- [ ] 各データセットの「バージョン管理」と「定期更新プロセス」を構築

**実装フェーズ**: WF#14 の開発準備段階
