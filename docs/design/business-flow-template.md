# THEBRANCH ビジネスフローテンプレート v1

## 概要

THEBRANCH のコアビジネスフロー：「アイデア → 実行可能な組織」への変換プロセス。
PM が立案したビジョンを、EM が組織化し、Engineer が実装する、3層の責任分離モデル。

---

## 標準フロー：ビジョン実行パイプライン

```
[PM] ビジョン立案
  ↓
[EM] 組織・チーム編成
  ↓
[Engineer] 実装実行
  ↓
[全員] フィードバック ← レビュー & 改善
```

---

## フェーズ別詳細フロー

### Phase 1: ビジョン立案（PM 責務）

**責務**: アイデアを実行可能な仕様に落とす

| ステップ | 入力 | 出力 | 期間 |
|---|---|---|---|
| **1.1 コンセプト定義** | ビジョン（文言） | コンセプト文書 | 30 分 |
| **1.2 要件定義** | コンセプト | 機能リスト（優先度付き） | 1 時間 |
| **1.3 成功指標定義** | 機能リスト | KPI（定量的指標） | 30 分 |
| **1.4 仕様書作成** | 全情報 | Markdown 仕様書 | 1～2 時間 |

**成果物**:
- 仕様書（Markdown）
- 機能リスト（優先度付き）
- 成功指標（KPI）

**意思決定基準**:
- ビジネス価値 vs. 実装コスト
- 優先度（Critical / High / Medium / Low）

---

### Phase 2: 組織・チーム編成（EM 責務）

**責務**: 仕様を実行可能な組織構造に変換

| ステップ | 入力 | 出力 | 期間 |
|---|---|---|---|
| **2.1 タスク分解** | 仕様書 | タスク一覧（DAG 依存関係） | 1～2 時間 |
| **2.2 チーム編成** | タスク一覧 | チーム定義（メンバー割り当て） | 1 時間 |
| **2.3 リソース計画** | タスク + チーム | 実装スケジュール（マイルストーン） | 1 時間 |
| **2.4 リスク評価** | 全情報 | リスク管理計画 | 30 分 |

**成果物**:
- タスク DAG（task-manager-sqlite 投入）
- チーム構成図
- 実装スケジュール（ガントチャート）
- リスク・緩和計画

**EM の判断ポイント**:
- 並行実行可能なタスク群の抽出
- クリティカルパスの特定
- リソース不足時の優先順位付け

---

### Phase 3: 実装実行（Engineer 責務）

**責務**: タスクを完了させ、成果物を作成

| ステップ | 入力 | 出力 | 期間 |
|---|---|---|---|
| **3.1 開発環境構築** | タスク一覧 | 稼働環境 | 1 時間 |
| **3.2 実装** | タスク定義 | コード / ドキュメント | 可変 |
| **3.3 ユニットテスト** | コード | テスト結果 + カバレッジ | 可変 |
| **3.4 統合テスト** | 複数機能 | 動作確認記録 | 可変 |

**成果物**:
- 実装コード
- テストコード
- PR（コードレビュー待ち）
- 実装ドキュメント

---

### Phase 4: レビュー & 改善（全員参加）

**責務**: 品質保証と継続改善

| ステップ | 責務 | チェック項目 |
|---|---|---|
| **4.1 コードレビュー** | Engineer / Tech Lead | 可読性、テスト網羅性、セキュリティ |
| **4.2 EM レビュー** | EM | スケジュール遵守、品質達成度 |
| **4.3 PM レビュー** | PM | 要件充足度、KPI 達成度 |
| **4.4 本番リリース** | All | デプロイ実行、監視開始 |

**成果物**:
- レビュー記録
- リリースノート
- 改善リスト

## タスク依存関係パターン集

### パターン A: シーケンシャル（直列）

```
Task A → Task B → Task C → Task D
```

**用途**: 前のタスクの成果物が後のタスクの入力になる場合
**例**: データベース設計 → マイグレーション → API 実装

### パターン B: パラレル（並列）

```
    ┌→ Task A
Task 0
    └→ Task B
       └→ Task C
```

**用途**: 独立したタスクを同時進行する場合
**例**: フロント実装 || バック実装 → 統合テスト

### パターン C: ファンアウト＆ファンイン（集約）

```
Task A ──┐
         ├→ Integration Test → Task D
Task B ──┤
         ├→ (統合)
Task C ──┘
```

**用途**: 複数の下流タスクが 1 つの上流タスクに依存
**例**: UI / DB / API が揃ったら E2E テスト

## チーム構成テンプレート

### 最小スタッフ（1～3 人）

```
PM 1 人（兼 EM）
  ├─ Engineer 1 人
  └─ Engineer 2 人
```

**特徴**:
- PM が直接 EM 役を兼務
- タスク分散が限定的
- 短期プロジェクト向け

### 標準スタッフ（4～8 人）

```
PM 1 人
  ├─ EM 1 人
  │   ├─ Engineer 1 人
  │   ├─ Engineer 2 人
  │   └─ Engineer 3 人（最大）
  └─ Tech Lead 1 人（オプション）
```

**特徴**:
- PM / EM / Engineer の役割明確化
- 並行実行最大3タスク
- 中期プロジェクト向け

### スケール構成（9 人以上）

```
PM 1 人
  ├─ EM 1 人（チーム A）
  │   ├─ Engineer A-1
  │   ├─ Engineer A-2
  │   └─ Engineer A-3
  ├─ EM 2 人（チーム B）
  │   ├─ Engineer B-1
  │   ├─ Engineer B-2
  │   └─ Engineer B-3
  └─ Tech Lead 1 人
```

**特徴**:
- 複数 EM で並行チーム管理
- タスク交差の最小化
- 大規模プロジェクト向け

## 意思決定フロー

### 優先度判定（PM → EM）

```
新規要件提案
  ↓
┌─────────────────────┐
│ 優先度ジャッジ        │
├─────────────────────┤
│ Critical? → Yes → 即実行
│ High? → Yes → 次週以内
│ Medium? → Yes → バックログ
│ Low? → YES → キューイング
└─────────────────────┘
```

### リスク評価（EM）

```
タスク受領
  ↓
リスク評価（技術的難度 × 実装期間）
  ↓
┌──────────────────────┐
│ リスク度判定          │
├──────────────────────┤
│ 高 → EM 直接検証
│ 中 → 定期進捗確認
│ 低 → 自律実行
└──────────────────────┘
```

### KPI 達成度判定（PM）

```
リリース後
  ↓
KPI 測定（プロダクト 1～4 週間）
  ↓
┌──────────────────┐
│ 達成度ジャッジ    │
├──────────────────┤
│ 100% 達成 → 次フェーズ
│ 70～99% → 改善実装
│ <70% → 原因分析 → ピボット
└──────────────────┘
```

## コミュニケーション・同期スケジュール

### 日次（Daily）

| 参加者 | 内容 | 時間 |
|---|---|---|
| Engineer | スタンドアップ（ブロッカー共有） | 15 分 |
| EM | チーム進捗確認 | 30 分 |

### 週次（Weekly）

| 参加者 | 内容 | 時間 |
|---|---|---|
| EM | PM との仕様・進捗調整 | 1 時間 |
| PM | マイルストーン確認 | 30 分 |

### 月次（Monthly）

| 参加者 | 内容 | 時間 |
|---|---|---|
| All | KPI レビュー & 改善討議 | 2 時間 |
| PM | 次月仕様・優先度確定 | 1 時間 |

## テンプレート活用例

### 例 1: 新機能「AI アシスタント統合」

**Phase 1: PM ビジョン立案（1 日）**
- コンセプト: 「エンジニアの開発時間を 30% 短縮」
- 要件: AI チャット UI + API 統合 + ナレッジベース
- KPI: チャット利用率 50%、解答満足度 4/5

**Phase 2: EM 組織化（2 日）**
- Task 1: ナレッジベース準備（1～2 日）
- Task 2: AI API 統合（3～5 日）
- Task 3: UI 実装（2～4 日）
- 並行実行: Task 1 || Task 2 → Task 3

**Phase 3: Engineer 実装（6 日）**
- EM が 3 人の Engineer を割り当て
- 定期的に進捗確認 & ブロッカー解除

**Phase 4: レビュー＆改善（1～2 週間）**
- コードレビュー
- 本番リリース
- KPI 測定 & 改善

## 補足：「翌朝から機能する組織」の実現

✨ **このテンプレートが保証する 3 つのこと**:

1. **明確な責任分離** — PM / EM / Engineer がそれぞれの役割を理解
2. **実行可能性** — 抽象的なアイデアが具体的なタスク DAG に変換される
3. **継続改善** — KPI レビュー & 改善ループで常に進化

これにより、個人のアイデアが 1 日～数週間で組織として機能する状態を実現できます。
