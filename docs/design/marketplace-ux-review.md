## 1. 総評

**marketplace-ux.md は高品質な仕様書**です。以下の点が優れています：

### ✅ 強み
- **構造が明確**: Context → フロー → 3画面詳細 → アルゴリズム → API → DB → チェックリストという論理的な流れ
- **実装準備が整っている**: API仕様、DB スキーマ、実装チェックリストがすぐに使える形で記載
- **ユーザー中心設計**: 「翌朝から動く」感を実現する具体的なUI/UX
- **3層構造のアルゴリズム詳細**: ビジョン解析 → スコアリング → ランキングの段階的な解説と計算例

### ⚠️ 改善必要箇所（優先度別）
1. **エッジケース処理の記載不足** - スコアが同点の場合、テンプレートなし、複数言語入力など
2. **パフォーマンス要件の欠落** - レスポンスタイム、キャッシュ戦略
3. **スコアリング精度検証方法が曖昧** - 既知テストケースの不足
4. **カスタマイズフロー完成度** - プロセス追加フローの詳細化が必要
5. **多言語・地域対応の検討不足** - 日本とグローバル対応
6. **既存システムとの連携不明確** - KuzuDB グラフ関係の活用方法

### 2.3 スコアリングアルゴリズム検証（スコア: 7.5/10）

**計算例の検証結果**:

Finance テンプレートの計算 (Section 4, Layer 2)
```
keyword_match_score = (95 + 80 + 50) / 3 = 75%  ✅ 正確
entity_match_score = (0 + 0) / 2 = 0%           ✅ 正確
intent_match_score = 15%                         ⚠️ 懸念: なぜ 15% なのか説明不足
最終スコア = 75 × 0.7 + 0 × 0.2 + 15 × 0.1
          = 52.5 + 0 + 1.5 = 54%               ✅ 正確
```

Sales テンプレートの計算
```
keyword_match_score = (100 + 85) / 2 = 92.5%    ✅ 正確
entity_match_score = 25%                         ⚠️ 計算根拠不明: 「月額スコア = 250 / 1000 = 25%」が
                                                    レンジ判定と整合しない
intent_match_score = 20%                         ⚠️ 改善: Finance の +15 より高い理由を説明
最終スコア = 92.5 × 0.7 + 25 × 0.2 + 20 × 0.1
          = 64.75 + 5 + 2 = 71.75% → 72%      ✅ 正確
```

**改善提案**:

```markdown
### スコアリング精度テストケース（実装時に追加すべき）

[テストケース 1] ビジョン "決算自動化" のスコアリング
入力: vision_input = "決算業務を完全自動化したい"
期待値:
  - Finance テンプレート: 85% 以上（スコア条件: キーワード "決算" weight=0.95）
  - Accounting テンプレート: 90% 以上
  - HR テンプレート: < 30%
実装チェック: 上記スコア範囲が得られることを確認

期待値: 複数テンプレート（Sales, Marketing, Product）が 50-70% 程度で並ぶ
理由: 「成長」は汎用キーワード → 複数候補を提示すべき

[テストケース 3] テンプレートなし "UFO デザイン部署立ち上げ"
```

### 2.4 ユーザージャーニー検証（スコア: 8.5/10）

3つのペルソナでの検証：

#### ペルソナ 1: Founder（初回ユーザー）
```
ジャーニー:
  Step 0: ビジョン入力 "営業チーム立ち上げ、月商1000万"
    → Step 1: テンプレート選択フロー表示 ✅ 推奨 92% が大きく表示
    → Step 2: 詳細確認（ロール、予算見積）✅ 意思決定に必要な情報有
    → Step 3: カスタマイズ ✅ メンバー数・予算スライダーで簡単調整
    → 実行

痛点分析:
  ✅ 「翌朝から動く」感を実現できる設計
  ⚠️ 初回ユーザーは "営業部長" が何をするのか不明確かもしれない
     → 提案: テンプレート詳細で "営業部長" の日次タスク例を 1-2 個表示
```

#### ペルソナ 2: Side-Gig Manager（副業・運営ユーザー）
```
ジャーニー:
  前提: 既に他の部署を運営しており、2つ目の部署立ち上げを考えている
  
  Step 0: ビジョン入力 "マーケティング部、ブランドを確立"
    → Step 1: テンプレート選択 ✅ 複数案が表示される
    → Step 2: 詳細確認
    → Step 3: カスタマイズ ✅ 予算が既存部署の 30% に抑えたい
             入力: 月額 150万円 → リアルタイム警告 ✅ 「市場相場比 -45%」
    → 判断: 「この部署は作らない」または「予算を増やす」

強み:
  ✅ リアルタイム予算警告で、実現不可能な設定を防止
  
課題:
  ⚠️ 既存部署との比較・共存戦略の提案がない
     → 提案: カスタマイズ画面に「組織内での優先度設定」を追加？
  ⚠️ 予算が低いときの代替案提示がない
     → 提案: 警告時に「月額 150万で実現可能なプロセス構成」を提示
```

#### ペルソナ 3: Organization Manager（組織拡大ユーザー）
```
ジャーニー:
  前提: 既に 3 部署を運営、4 つ目を立ち上げ
  
  Step 0: ビジョン入力 "法務部、コンプライアンス強化"
    → Step 1: テンプレート選択 ✅ HR・General Management が候補
    → Step 2-3: テンプレート比較 ✅ [比較を見る] で複数候補を横並び比較
    → カスタマイズ: 既存組織標準（月額 300万/部署）に合わせる

強み:
  ✅ 複数テンプレート比較機能で意思決定が容易
  ✅ リアルタイム予算計算で、組織全体の予算配分が可能

課題:
  ⚠️ 既存部署との連携・依存関係の設定ができない
     例: 「法務部は営業推進部と連携→営業契約レビュー」
  ⚠️ オーケストレーション（複数部署の同期・優先度）の概念がない
```

**ペルソナ検証総括**:
- Founder: ✅ 十分対応可能
- Side-Gig Manager: ⚠️ 予算制約時の代替案提示が必要
- Organization Manager: ⚠️ 複数部署の連携・共存戦略の仕組みが足りない

### 2.7 既存システムとの連携（スコア: 6/10）

⚠️ **懸念点が多い**:

| 既存システム | 接続状況 | 懸念 |
|------------|--------|-----|
| **KuzuDB** | 記載なし | テンプレート間の依存関係（法務部が営業部に依存等）の実装方法が不明 |
| **オーケストレーター** | 記載なし | 複数部署起動時の優先度・スケジューリングをどう制御するか |
| **ダッシュボード UI** | 記載あり（画面遷移）| OK - Step 3 後にダッシュボード自動遷移 |
| **認証・権限** | 記載なし | 複数部署が存在する場合、ユーザーがどの部署を操作するかの選択画面が必要か |

**提案**:
```markdown
## 既存システム連携ガイドライン（追加セクション）

### KuzuDB との統合
テンプレート間の依存関係を グラフで管理
  Sales テンプレート
    ↓ (需要予測のため)
  Forecasting テンプレート
    ↓ (財務分析のため)
  Finance テンプレート

実装方法:
  - テーブル: template_dependencies (template_id, depends_on_template_id)
  - オンボーディング時: 関連テンプレートも提案に含める
  - UI: "このテンプレートは Sales テンプレートと連携します" と表示

### 複数部署のオーケストレーション
複数部署が同時起動する場合の制御
  - 優先度設定: カスタマイズ画面で優先度を設定
  - スケジューリング: 1 つずつ起動 vs 並行起動の選択
  - リソース管理: 総予算・総メンバーの制約チェック

### 権限管理
ユーザーが複数部署を持つ場合
  - 部署切り替え UI: ダッシュボード左サイドバーに部署リスト
  - オンボーディング URL: 新規部署作成時のディープリンク
```

```markdown
### 2. 予算計算のエッジケース

[ケース] 予算がテンプレート min_members の給与合計より低い
  例: 月額 50万円、営業部長給与 120万/月が必須
  現在の処理: 警告表示のみ
  提案実装:
    - 警告レベル: エラー（赤）
    - メッセージ: "この予算では実現不可能です"
    - [実現可能な予算を計算] ボタン
    - 最小予算の提示

[ケース] メンバー数スライダーで min/max 制約違反
  例: 営業部長 1 名必須だが、total_members = 0
  現在の処理: 未定義
  提案実装:
```
