## 1. 目的

THEBRANCH では、タスク管理における定型的な流程（バグ報告 → 割り当て → 実装 → レビュー → 完了）を繰り返す。本機能は、このような標準フロー・組織固有のフロー・プロジェクト固有のフロー を**テンプレート化** し、新規タスク作成時に選択肢として提供することで、以下を実現する：

- **再利用性の向上**: 定型フローの再定義・再入力を排除
- **一貫性の確保**: 複数プロジェクト間でフローを標準化
- **効率化**: テンプレート選択 → タスク一括生成で初期設定時間を短縮
- **スケーラビリティ**: ワークフロー定義の柔軟な管理

---

## 2. スコープ

### 2-1. 対象機能

#### A. ワークフローテンプレート定義

- **テンプレート作成**: 定型フローを再利用可能なテンプレートとして定義
- **テンプレート管理**: テンプレートの作成・編集・削除・バージョン管理
- **フェーズ設計**: 複数フェーズをシーケンシャル・パラレルで定義
- **スペシャリスト割り当て**: フェーズごとに必要なロール（PM・Engineer・QA・DevOps 等）を定義

#### B. テンプレート選択 UI

- **テンプレート一覧表示**: 利用可能なテンプレートを表示
- **テンプレート詳細表示**: フェーズ・タスク数・見積もり時間等を表示
- **フィルタリング**: ロール・プロジェクト別にテンプレートをフィルタリング

#### C. ワークフローインスタンス化

- **インスタンス作成**: テンプレート選択 → Specialist 指定 → インスタンス化
- **自動タスク生成**: インスタンス化時に、定義されたタスクを一括生成
- **順序制御**: フェーズ間の依存関係・順序を自動管理

### 2-2. 非対象（今後の拡張）

- リアルタイム並行実行の高度な制御（e.g., 動的リバランシング）
- テンプレート間の継承・混合
- ワークフロー実行中のダイナミック修正
- AI による自動ワークフロー学習・最適化

## 3. 要件詳細

### 3-1. 機能要件（FR）

#### FR1: ワークフローテンプレートの定義

| # | 要件 | 詳細 |
|---|---|---|
| FR1.1 | テンプレート作成 API | テンプレート名・説明・バージョンを指定して新規作成 |
| FR1.2 | フェーズ定義 | 各テンプレートに複数フェーズを定義可能（最大 10 フェーズ） |
| FR1.3 | タスク定義 | 各フェーズ内にタスク定義を保持（最大 50 タスク） |
| FR1.4 | Specialist 割り当て | フェーズごとに必要なロールと人数を定義 |
| FR1.5 | 順序制御 | フェーズ間の順序（sequential / parallel）を定義 |
| FR1.6 | バージョン管理 | テンプレートのバージョン履歴を保持 |
| FR1.7 | ステータス管理 | draft / active / deprecated の 3 ステータス |

#### FR2: テンプレート選択 UI

| # | 要件 | 詳細 |
|---|---|---|
| FR2.1 | テンプレート一覧表示 | 利用可能なテンプレートをリスト表示 |
| FR2.2 | テンプレート詳細表示 | 選択したテンプレートのフェーズ・タスク・見積もり時間を表示 |
| FR2.3 | フィルタリング | ロール・ステータス・プロジェクト別にフィルタリング可能 |
| FR2.4 | 検索機能 | テンプレート名・説明で検索可能 |
| FR2.5 | プレビュー機能 | インスタンス化前にフェーズ・タスク構造をプレビュー可能 |

#### FR3: ワークフローインスタンス化

| # | 要件 | 詳細 |
|---|---|---|
| FR3.1 | インスタンス作成 | テンプレート選択 → Specialist 指定 → インスタンス化 |
| FR3.2 | Specialist 割り当て UI | フェーズごとに適切な Specialist を選択可能 |
| FR3.3 | 自動タスク生成 | インスタンス化時に dev_tasks テーブルにタスクを自動作成 |
| FR3.4 | 依存関係設定 | タスク間・フェーズ間の依存関係を自動設定 |
| FR3.5 | 順序制御 | phase_order に基づいてフェーズの実行順序を制御 |

#### FR4: テンプレート管理

| # | 要件 | 詳細 |
|---|---|---|
| FR4.1 | テンプレート編集 | 作成済みテンプレートの編集（draft/active の distinction） |
| FR4.2 | テンプレート削除 | 不要になったテンプレートを削除（ただし active テンプレートは削除不可） |
| FR4.3 | テンプレート複製 | 既存テンプレートをベースに新規テンプレート作成 |
| FR4.4 | テンプレート共有 | プロジェクト・組織間でテンプレートを共有可能 |

### 3-2. 非機能要件（NFR）

| # | 要件 | 詳細 | 目標値 |
|---|---|---|---|
| NFR1 | パフォーマンス | テンプレート一覧表示の応答時間 | < 500ms |
| NFR2 | スケーラビリティ | 最大テンプレート数 | 1000+ |
| NFR3 | 同時ユーザー数 | 同時インスタンス化操作 | 100+ concurrent requests |
| NFR4 | データ整合性 | トランザクション処理 | ACID 準拠（SQLite） |
| NFR5 | セキュリティ | Specialist 割り当ての権限確認 | role_based_access_control |
| NFR6 | 監査ログ | 操作履歴の記録 | テンプレート作成・編集・削除・インスタンス化の記録 |

### 3-3. ユースケース

#### UC1: プロダクトローンチテンプレートを選択してワークフロー実行

```
1. PM がテンプレート一覧から「Product Launch」を選択
2. システムが「Product Launch」テンプレートの詳細を表示
   - Phase 1: Planning（PM）
   - Phase 2: Development（Engineer × 3）
   - Phase 3: QA（QA Engineer）
   - Phase 4: Launch（DevOps）
3. PM が各フェーズの Specialist を指定（例: Alice for Planning, Bob-Tom for Dev, Carol for QA, Dave for DevOps）
4. システムがワークフロー instance を作成し、自動的にタスクを生成
5. 各 Specialist に割り当てられたタスクが dev_tasks に登録される
6. Phase 1 から実行開始
```

#### UC2: バグ修正ワークフロー（簡易）

```
1. QA Engineer がバグ報告 UI から「Quick Bug Fix」テンプレートを選択
2. テンプレートが以下フェーズを定義:
   - Phase 1: Triage（QA）
   - Phase 2: Fix（Engineer）
   - Phase 3: Verification（QA）
3. QA Engineer がフェーズごとの Specialist を指定
4. ワークフロー instance が作成され、3 つのタスクが自動生成
5. Phase 1 → Phase 2 → Phase 3 の順序で実行
```

## 4. データモデル（概要）

### 4-1. テーブル構成

| テーブル | 役割 |
|---|---|
| `workflow_templates` | テンプレートのメタデータ（名前・説明・ステータス） |
| `wf_template_phases` | テンプレート内のフェーズ定義 |
| `wf_template_tasks` | 各フェーズ内のタスク定義 |
| `workflow_instances` | テンプレートのインスタンス |
| `workflow_instance_specialists` | インスタンス内の Specialist 割り当て |
| `wf_instance_nodes` | インスタンス内のフェーズ実行状態 |
| `dev_tasks` | 実際のタスク（既存テーブル） |

詳細は `design/WORKFLOW-TEMPLATE-MODEL.md` を参照。

## 5. API/UI 設計（概要）

### 5-1. REST API

#### テンプレート管理

```
GET    /api/workflow-templates              # テンプレート一覧取得
POST   /api/workflow-templates              # テンプレート作成
GET    /api/workflow-templates/{id}         # テンプレート詳細取得
PUT    /api/workflow-templates/{id}         # テンプレート編集
DELETE /api/workflow-templates/{id}         # テンプレート削除
```

#### インスタンス管理

```
POST   /api/workflow-instances              # インスタンス化（テンプレート選択 → Specialist 割り当て）
GET    /api/workflow-instances/{id}         # インスタンス詳細取得
GET    /api/workflow-instances/{id}/tasks   # インスタンス内のタスク一覧
```

詳細は `design/WORKFLOW-TEMPLATE-DESIGN.md` を参照。

### 5-2. UI フロー

#### テンプレート選択画面

```
┌─ テンプレート一覧
│  ├─ [フィルタ: ロール / ステータス]
│  ├─ [検索: テンプレート名]
│  └─ テンプレートリスト
│     ├─ "Product Launch" (v1.0)
│     ├─ "Quick Bug Fix" (v2.1)
│     └─ ...
└─ テンプレート詳細表示
   ├─ Phase 1: Planning
   ├─ Phase 2: Development
   └─ [Specialist 割り当て] → [インスタンス化]
```

詳細は `design/WORKFLOW-TEMPLATE-DESIGN.md` を参照。

## 8. 依存関係・リスク

### 依存関係

- **task-manager-sqlite スキル**: workflow_templates/phases/tasks テーブル実装
- **dev_tasks テーブル**: 自動タスク生成のターゲット
- **agents テーブル**: Specialist 選択肢の取得

### リスク

| リスク | 対策 |
|---|---|
| フェーズ間の順序制御が複雑化 | テスト駆動で実装・BDD テスト導入 |
| 大規模テンプレートでの性能低下 | インデックス設計・クエリ最適化 |
| Specialist 割り当てロジックの誤り | 統合テスト・ユースケース検証 |
